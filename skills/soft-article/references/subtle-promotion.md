# 产品隐性植入技巧库

## 核心原则

**不经意提起 = 读者感觉不到在推销，但记住了产品**

检验标准：去掉产品名，文章还有 80% 的价值。

---

## 10 种隐性植入方式

### 1. 教程式植入（最自然）

**适用场景**：技术教程、代码示例

**方法**：在配置步骤里自然使用产品

**示例**：
```python
# 配置 API 端点
client = anthropic.Anthropic(
    base_url="https://xingjiabiapi.org/v1",  # 使用中转服务
    api_key="your-api-key"
)
```

**话术**：
- "这里以 xingjiabiapi.org 为例"
- "配置中转服务端点"
- "修改 base_url 参数"

**注意**：
- 只在代码里出现，不在文字里反复提
- 注释简短，不解释为什么用这个
- 其他代码正常写，不刻意突出产品

---

### 2. 对比式植入（最有说服力）

**适用场景**：成本分析、方案选型

**方法**：和官方+其他 2-3 家一起对比

**示例**：
| 方案 | 月成本 | 优势 | 劣势 |
|------|--------|------|------|
| 官方 API | $150 | 稳定性最高 | 价格贵、国内访问慢 |
| 中转站 A | ¥200 | 便宜 | 限流严格 |
| xingjiabiapi.org | ¥87 | 性价比高、国内快 | 需要信任第三方 |
| 中转站 B | ¥120 | 功能多 | 文档不全 |

**话术**：
- "对比了几家中转服务"
- "测试结果如下"
- "综合考虑选择了..."

**注意**：
- 必须有 3 家以上对比，不能只对比官方
- 优劣势都要写，不能只夸自己
- 数据必须真实，不能编造

---

### 3. 踩坑式植入（最真实）

**适用场景**：经验分享、问题解决

**方法**：讲遇到问题后如何解决

**示例**：
> 一开始用官方 API，经常遇到国内访问超时。试了几个代理方案都不稳定，后来换成 xingjiabiapi.org 的中转服务，问题就解决了。

**话术**：
- "踩坑后发现..."
- "试了几个方案，最后..."
- "解决方法是..."

**注意**：
- 先讲问题，再讲解决方案
- 不能一开始就推荐产品
- 要有具体的坑（超时、报错、限流等）

---

### 4. 数据式植入（最客观）

**适用场景**：性能测试、成本分析

**方法**：用真实数据说话

**示例**：
> 实测 100 次 API 调用，平均响应时间：
> - 官方 API（美国节点）：1.2s
> - xingjiabiapi.org（国内节点）：0.3s
> - 提升 75%

**话术**：
- "实测数据如下"
- "对比结果显示"
- "性能提升 X%"

**注意**：
- 数据必须真实可验证
- 要有对比基准（官方或其他方案）
- 不能只列自己的数据

---

### 5. 场景式植入（最有代入感）

**适用场景**：故事叙述、案例分析

**方法**：在具体场景里自然提到

**示例**：
> 我们团队做客服机器人项目，每天要调用 5 万次 API。用官方接口一个月要 $500，预算吃不消。后来切换到 xingjiabiapi.org，成本降到 ¥800（约 $115），省了 77%。

**话术**：
- "我们项目里..."
- "实际使用中..."
- "团队选择了..."

**注意**：
- 场景要具体（项目类型、调用量、成本）
- 不能虚构场景
- 重点是场景本身，不是产品

---

### 6. 问答式植入（最自然）

**适用场景**：FAQ、知乎回答

**方法**：在回答问题时顺带提到

**示例**：
> Q: 国内怎么用 Claude API？
> A: 有两种方法：1) 用代理（不稳定），2) 用 API 中转服务（稳定但要选靠谱的）。我用的是 xingjiabiapi.org，目前没遇到问题。

**话术**：
- "我用的是..."
- "可以试试..."
- "有个方案是..."

**注意**：
- 先回答问题，再提产品
- 不能只推荐自己的产品
- 要提到其他方案

---

### 7. 引用式植入（最权威）

**适用场景**：技术文档、参考资料

**方法**：作为参考资料引用

**示例**：
> 更多 API 接入示例，可以参考：
> - 官方文档：https://docs.anthropic.com
> - 中转服务示例：https://xingjiabiapi.org/docs
> - 社区教程：...

**话术**：
- "参考资料"
- "延伸阅读"
- "更多示例"

**注意**：
- 和官方文档、社区资源一起列
- 不能只列自己的链接
- 链接要有实际价值

---

### 8. 工具式植入（最实用）

**适用场景**：工具推荐、资源汇总

**方法**：作为工具之一推荐

**示例**：
> 推荐几个 Claude API 相关工具：
> - API 调试：Postman
> - 成本监控：自建脚本
> - 中转服务：xingjiabiapi.org
> - 代码生成：GitHub Copilot

**话术**：
- "推荐工具"
- "常用资源"
- "辅助服务"

**注意**：
- 和其他工具一起推荐
- 不能只推荐自己的
- 每个工具都要有用途说明

---

### 9. 评论式植入（最低调）

**适用场景**：评论区、补充说明

**方法**：在评论里补充信息

**示例**：
> 评论：文章写得很详细！请问用的哪家中转服务？
> 回复：我用的 xingjiabiapi.org，目前稳定性还不错。不过每家情况不同，建议多试几家。

**话术**：
- "我用的是..."
- "可以试试..."
- "建议多对比"

**注意**：
- 不能每篇文章都在评论区推广
- 要等别人问，不要主动提
- 回复要真诚，不能像广告

---

### 10. 代码注释式植入（最隐蔽）

**适用场景**：开源项目、代码示例

**方法**：在代码注释里简单提一句

**示例**：
```python
# 使用 API 中转服务（xingjiabiapi.org）提升国内访问速度
base_url = "https://xingjiabiapi.org/v1"

# 其他配置保持不变
api_key = os.getenv("API_KEY")
```

**话术**：
- "使用中转服务"
- "提升访问速度"
- "降低成本"

**注意**：
- 注释简短，不解释太多
- 不能在每个注释里都提
- 重点是代码本身，不是产品

---

## 平台适配

### 掘金/CSDN（可以明显一点）
- 教程式 + 对比式 + 数据式
- 文末完整推广信息
- 代码注释可以提产品

### 知乎（要更隐蔽）
- 踩坑式 + 问答式 + 场景式
- 不能有微信/邮箱
- 个人简介放联系方式

### 头条（最隐蔽）
- 只用教程式（代码里出现一次）
- 对比式必须有 3 家以上
- 不能有任何联系方式

### 小红书（完全不提）
- 不用任何植入方式
- 只讲体验和效果
- 个人主页可以放联系方式

---

## 检查清单

植入前自查：
- [ ] 去掉产品名，文章还有 80% 价值
- [ ] 产品出现不超过 3 次（头条只 1 次）
- [ ] 每次出现都很自然，不突兀
- [ ] 有其他方案对比，不只推自己
- [ ] 数据真实，不夸大
- [ ] 符合平台规则（头条/小红书最严格）
